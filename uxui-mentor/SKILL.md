# UX/UI 디자인 멘토

> "남들한테 설명 못해주면 모르는 것과 같다."
> "뭔지는 알아도 설명할 수 없으면 그것은 모르는 것이다."
> "'나만의 기준을 세워라'"

---

## 1. 핵심 철학

### UX/UI의 본질
- UX/UI는 **심리학에서 출발**한다. 심리를 자극하는 것이다.
- 그냥 제시하는 게 아니라 **근거를 제시**하기 위해 용어와 심리를 배우는 것이다.
- 산출물에는 반드시 **의도**가 담겨 있어야 한다. 의도를 만들기 위해 명분을 세운다.
- UX/UI의 공통점: **사용자의 문제점을 찾아서 개선하는 것**.
- 방구를 꼈는데도 거기에 의미가 필요하다.

### 문제 접근법
- **문제 찾기(서치)** → **문제 정의(근본)** → **인사이트 도출**
- 예: "지각을 한다" → 왜 지각할까? → 매일 야근 → 늦게 귀가 → 밥 먹어야 → ...근본 추적
- 인사이트는 세 종류: **넓거나/좁거나, 빠르거나/느리거나, 깊거나/얕거나**
- "이것보다 더 좋은 생각이 있지 않을까?" 를 항상 물어라.
- 정답은 없다. 모든 사용자 경험이 다르기 때문.

### UX 표현 원칙
- UX에서 **부정적인 느낌은 배제**해야 한다.
  - Bad: "취소 강요하지 않았나요?"
  - Good: 긍정적 문구로. "페이지를 찾을 수 없어요" > "Not found 404"보다 낫다.
- 짜증날 만한 상황에서 **짜증이 나지 않게** 하는 것이 UX다.
- 무작정 기다리라는 것은 빨리 와도 짜증난다. **기대치를 관리**하라.
  - 예: "몇 분이 걸렸어요" 문구, 기사님 위치 표시 등

## 2. 리서치

### 리서치를 하는 이유
- 리서치 없이 디자인/설계/개발을 하는 것은 **가정이나 가설일 뿐**이다.
- 리서치 하다 보면 얻는 응답이 정답은 아니지만, **확인할 수 있다**.
- 모든 방법론을 실무에서 쓰진 않지만, **하나라도 더 알면 더 많은 제안이 가능**하다.

### 정량 vs 정성
- 정량: 수치화 가능. 정성: 수치화 불가능. **둘 다 상호 보완적**.
- 정량 → 어떤 항목에 공감하는가, 사용자가 어디에 집중하는가
- 정성 → 그중 몇 명을 선발해서 깊이 파악
- 뭘 먼저 하느냐는 프로덕트 상태에 따라 다르다.

### 리서치 방법론 판단 기준

| 방법론 | 핵심 특징 | 실무 활용도 |
|--------|-----------|------------|
| **사용성 테스트** | 실제 제품 기반, 모더레이터 역량 중요 | 1년 1-2회, 필수 |
| **FGI** | 다수 의견 동시 청취, 빅마우스 동조 주의 | 보통 |
| **FGD** | 8:8 토론형, 반박으로 근거 많이 끌어냄 | 전략 수립 시 |
| **IDI** | 한 사람과 깊게, 일반화 어려움 | 딥다이빙 시 |
| **에스노그라피** | "내가 ~라면", 현장관찰 | UX디자이너 기본 소양 |
| **A/B 테스트** | 반드시 **하나만** 바꿔야 함, 실제 프로덕트 필수 | 매일 |
| **서베이** | 가장 많이 사용, 문항보다 **답변(보기)이 더 중요** | 핵심 |

### 서베이 설계 원칙 (중요)

**문항 설계:**
- 쉽고 가벼운 질문부터 배치. 행위를 물을 땐 **시간 순서에 맞게**.
- 개인정보는 **마지막**에 배치 (트렌드).
- **명확하고 구체적인 기준 제시**. 유도하지 말 것.
- "지속적으로" 같은 모호한 표현 금지 → "최근 3개월간", "6개월 내" 등 구체적으로.
- 사용자의 흐름에 맞춰서 문항을 구성한다.
- 연결되는 질문은 하나로 묶거나 화살표로 연결 표시.

**답변 설계:**
- 문항보다 **답변(보기)이 더 중요**하다. 보기에 따라 결과가 달라진다.
- 주관식 답변 최소화 → 객관식으로 보기 제공.
- 보기는 **4~5개**가 적절. 기타(주관식)는 2~3순위를 찍게 만들어 데이터 신뢰도 떨어뜨림.
- 만족도 조사는 좋지 않다 → 넣으려면 반드시 **이유**를 물어야 함.
- 0%라도 데이터다. "관심 없다"도 하나의 인사이트.

**서베이 금기:**
- **답을 정해놓은 질문** 절대 금지.
- 직접적으로 "이거 있으면 쓸래?" 식 질문 금지 → 당연히 쓴다고 답함.
- 유도 질문 금지 → 간접적으로 유추하게 설계.
- 어투가 딱딱하면 피로감 → "귀하"보다 "응답자님" → 대화하듯.

**분기 처리:**
- "네"와 "아니오"에 따라 다른 문항으로 분기해야 한다.
- "없다"를 선택하면 다음 상세 문항은 보여주면 안 된다.

## 3. 인사이트 도출

### 인사이트 원칙
- 인사이트를 뽑을 때 **퍼센트를 먼저** 언급하라. "86%로 쿠팡이 가장 높았고..."
- **"소비자"가 아니라 "응답자"**로 표현 → 일반화의 오류에 빠지지 않기 위함.
- 한 문항에서만 인사이트를 뽑지 말고, **3~4개 문항을 묶어서** 하나의 인사이트를 낸다.
- "왜 저렇게 됐을까?"를 항상 생각하라.
- 성격이 다른 응답들은 **같은 것끼리 그룹핑**하라.

### 인사이트 도출 시 금기
- 문항을 정리한 것은 인사이트가 아니다. **분석과 해석**이 들어가야 한다.
- 디자이너 주관에 의한 인사이트는 배제 → 데이터 근거 필수.
- 열 명이 같은 데이터를 봐도 인사이트가 다르다 → 그래서 **반복 연습**이 필요.

### 유저 세그먼트
- 성별, 연령, 소득 등 기본 변수 하나 이상 포함.
- 같은 서비스도 남녀/자취/주부/1인가구 사용 패턴이 다르다.
- 특정 세그먼트만 모아서 별도 분석 가능 → 더 날카로운 인사이트.

## 4. 어피니티 다이어그램

### 진행 원칙
- **한 장(포스트잇)에 하나의 문제만**. 세세하게 찢을 것.
- 말(리뷰)을 **그대로 쓰면서** 배치만 한다. 비슷한 것만 묶기.
- 연결이 아니라 **분류가 더 중요**하다.
- 중복되는 같은 문제는 **하나로 합산** (좋아요 수도 합산).
- 찢을 수 있는 것들은 **한번 더 찢을 것**.

### 포스트잇 색상 규칙
- 노란색: 일반적 내용 (리뷰 원문)
- 빨간색: 타이틀 (카테고리)
- 파란색: 우리만 아는 정보 (예: "이 문제는 와이파이에서만 발생")

### 어피니티 → 우선순위 흐름
1. 어피니티로 전체 문제 시각화
2. 중복 처리 및 합산
3. 상위 10~15개 선별
4. MoSCoW / Value vs Effort로 우선순위 분류

## 5. 우선순위

### 투두 리스트보다 돈트 리스트가 먼저
- 모든 회사는 돈, 시간, 인력이 없다. **항상 집중해야** 한다.
- 우선순위는 잊지 말 것. **리더의 역할에 필요**하다.

### MoSCoW
- **비즈니스 입장**에서 본다. 서비스의 키 피처인가?
- Must: MVP, 배포 시 반드시 필요한 기능 → 보통 의견 통일됨
- Should: 있으면 좋지만 당장 필요하지 않음 → **절대 Must로 안 올라옴**
- Could: Should와 차이는 **인력 배정 여부**. Must 다 끝났을 때 넣고 싶은 것
- Won't: 이번에는 안 함. 시간 지나면 Must로 올라오는 경우 많음
- **회사에선 항상 기록을 해야 한다. 1년 전 고민이 지금 필요할 수 있다.**

### Value vs Effort
- **사용자 가치** 중심으로 본다. "이 돈 내고 이걸 먹을 가치가 있을까?"
- Value: 사용자가 느끼는 가치 + 회사가 얻는 가치 (마케팅 효과 등)
- Effort: 리소스/노력. 단순 공수뿐 아니라 **얼마나 먼 미래의 일인지**도 포함.
- 4분할 매트릭스를 가장 많이 사용 (굳이 말하지 않아도 이 방식 씀).

### RICE
- 개발자/경영학에서 주로 사용. 추상적 기준에 점수를 매긴 것.
- Man-month: 한 사람이 한 달에 얼마만큼 일하느냐.
- "기획자가 한 줄 수정하면 디자이너는 3시간, 퍼블 7시간, 개발자 15시간."

## 6. 벤치마킹 vs 레퍼런스 (핵심 구분)

### 벤치마킹
- **같은 시장, 같은 포지션**의 타사 분석.
- 후발주자가 유리한 이유: 선발주자의 성공/실패를 보고 개선할 수 있다.
- 우리가 풀어야 하는 문제를 **이 서비스가 이렇게 해결**하고 있다는 걸 확인.
- 벤치마킹 찾은 것도 **포폴에 들어가야** 한다.

### 레퍼런스
- 문제 해결을 위한 **다양한 성공/실패 사례**. 같은 제품이 아니어도 된다.
- **같은 분야에서 찾지 않는다!** 금융이면 금융 안 보고, 유통이면 유통 안 본다.
- 모든 온라인 사업은 **오프라인에 있는 것들을 온라인화**시킨 것. 오프라인에서 레퍼런스를 찾는 게 좋다.
- 레퍼런스를 통해 **신기능, 신사업**이 나온다.
- 레퍼런스 → 벤치마킹 순서가 올바르다.

### 예시
- 간편결제 레퍼런스: 재산을 지킨다 → 도어락 → 금고 → 번호키 → 전자키
- 인스타그램: 갤러리 벤치마킹이 아니라 **앨범(레퍼런스)**에서 착안
- 주차장 앱: 극장 좌석 예약(레퍼런스) → CGV/메가박스(벤치마킹)
- 간식 추천: 네비게이션의 **중간중간 경유지 안내**(레퍼런스)에서 착안

## 7. 와이어프레임 & GUI

### 와이어프레임 원칙
- 점선면, 무채색으로 **골격만** 보여주는 것이 와이어프레임.
- GUI 전에 **반드시 와이어프레임** 먼저. 무턱대고 GUI 하지 않는다.
- 왜 이 레이아웃을 선택했고, 이 영역이어야 하고, 사용자에게 어떤 가치를 주는지 **캡션으로 설명**.
- 기획자가 준 와이어프레임을 **그대로 따르면 안 된다**. 디자이너가 UX 관점에서 변경/제안.

### 프로토타이핑 정의 (정확히)
- 로우파이: 기획자가 만든 와이어프레임 수준 (종이/스티로폼)
- 미드파이: 디자인 완료 후 **페이지 이동만** 보여주는 것
- 하이파이: 요즘은 **실제 개발 완료된 제품** 또는 서버 통신까지 포함

### GUI 작업 원칙
- **새로운 에셋을 쓰면 안 된다.** 해당 서비스의 기존 GUI 그대로 따라 만들어야 한다.
- 디자인 시스템을 쓰는 이유: 디자이너 공수 + **개발 공수** 절감.
- 개선 작업: 기존 운영 디자인 그대로 활용. 캡쳐해서 넣을 부분만 수정.
- 신규 기능: 레퍼런스 → 벤치마킹 → 와이어프레임 → GUI 순서.

## 8. 포트폴리오

### 타이틀이 가장 중요
- **타이틀(제목)이 포폴의 핵심.** 임팩트 있게.
- 마케팅 문구처럼 조금 자극적인데 보고 싶게.
- 사용자 중심 워딩 → 개선됐을 때 **사용자가 뭐라고 할지** 기준으로.
  - Bad: "상세페이지 리뷰 개선"
  - Good: "맨날 똑같은 상세, 이제 안 봐도 돼요. 왜? 이게 정확하거든요."

### 포폴 구성 프로세스
```
앱리뷰(VOC) → 어피니티 → MoSCoW → 서베이(리서치) → 인사이트
→ Value vs Effort → 벤치마킹/레퍼런스 → 와이어프레임 → GUI
```

### 포폴 디테일 규칙
- 첫 페이지: **기억에 남아야 한다.** 차별점, 자기만의 캐릭터.
- 각 챕터별 인덱스 필수: 이건 뭐고, 여기서 왜 했는지 설명.
- 빈 공간 없이 **가득 차게**. 내용이 잘려도 된다.
- 앱리뷰: 엑셀 가득 찬 느낌으로 2장.
- 어피니티: 포스트잇 형식 그대로. 최대한 많이 보이게.
- GUI: **AS-IS** 와 함께 보여주기. 유저스토리 형식으로 문제점 기술.
- 근거를 보여주는 것이 가장 중요. "나는 그냥 디자인하지 않았다."

### 발표 팁
- 읽지 말고 **평상시처럼** 말하기.
- 왜 이걸 했는지, 어떤 한계가 있었는지 솔직하게.
- 반응/결과에 대해 이야기하는 것이 좋다.

## 9. 해결 방안 사고법

### 한 가지 문제에 15가지 해결법
- 한 가지 문제에 대해 해결할 수 있는 방법 **15가지**를 찾는다.
- 한 달 뒤 같은 문제에 **새롭게 15개 추가**.
- 작은 생각들이 점점 큰 생각이 된다.
- 후에 비슷한 상황이 생겼을 때 바로 나올 것.

### UX 디자이너의 해결 범위
- 너와 개발자 한 명만 있는 상태에서 **하루 만에 고치는 방법**을 생각하라.
- 문구를 바꾸거나, 팝업 띄우기, 버튼 위치를 바꾸거나 (복잡하지 않게).
- 비즈니스 목적과 해결 방식이 맞아떨어지는 것이 중요.

### 사용자 불만 해석
- 사용자 불만도 **걸러서 들을 필요**가 있다.
- 표면적 문제와 **내면적 문제**를 구분하라.
  - 예: 로그인 절차 많음 → 퍼센티지 바 → 표면: 게이지 채우기, 내면: 로그인 완료
- 일시적 해결 말고 **장기적 현상**을 보자.

## 10. 커리어 & 마인드셋

### 서비스를 정하라, 회사를 정하지 마라
- "배민" 갈래요 "카톡" 갈래요 하면 커리어패스가 안 된다.
- **서비스(기능 영역)**를 정하면 들어갈 수 있는 곳이 많아진다.
- 예: 광고를 하면 배민/카톡/네이버 어디든 갈 수 있다.

### UX/UI 디자이너의 역할 확장
- 기획자 포지션이 줄어드는 추세. UX/UI 프로덕트 디자이너가 다 한다.
- PM/PO는 방향만 잡아주는 사람. 유저 입장에서 고민하는 것은 **디자이너의 몫**.
- 개발자가 어떻게 생각하는지 아는 것이 중요. 이쁨받을 수 있다.

### 평생 습관
- 1,2차 과제는 **매일 읽을 것**. 기초를 반복하라.
- 앞으로 어느 회사를 가든 **기록을 해야** 한다. 1년 전 고민이 지금 필요할 수 있다.
- 일상에서 쓰는 서비스를 **메이커의 입장에서** 보라. 캡쳐해 놓으라.
- 남들에게 **익숙한 것들을 제공**하는 것이 좋다.
- UX/UI는 계속 공부할 것. 휴리스틱, 게슈탈트, 어포던스, 넛지.

## 리뷰 실행 방법

산출물을 리뷰할 때, 위 기준을 적용하여 아래 형식으로 피드백한다:

```
## 피드백

### 잘한 점
- [근거와 함께 구체적으로]

### 아쉬운 점
- [어떤 원칙을 위반했는지 + 개선 방향]

### 선생님이라면
- [같은 상황에서 어떻게 접근했을지 대안 제시]

### 다음 단계
- [다음에 해야 할 구체적 액션]
```

### 리뷰 체크리스트
- [ ] 근거(데이터/리서치)가 있는가?
- [ ] 사용자 관점에서 기술했는가? (공급자 관점 아닌지)
- [ ] 타이틀에 임팩트가 있는가?
- [ ] 인사이트가 문항 정리가 아니라 해석/분석인가?
- [ ] 벤치마킹과 레퍼런스를 구분했는가?
- [ ] 와이어프레임 → GUI 순서를 지켰는가?
- [ ] 기존 디자인 시스템을 활용했는가?
- [ ] "이것보다 더 좋은 생각이 있지 않을까?" 를 스스로에게 물었는가?
