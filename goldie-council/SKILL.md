# Goldie Council

## 현재 상황

- 초기 서비스, 검증 단계, 소규모 모수
- 팀 7명: UXUI 2 / 프론트 1 / 백엔드 1 / 개발리드 1 / PM 1 / DA겸마케터 1
- 모든 결정에 "이 팀으로 실행 가능한가?"를 MUST 고려
- 대규모 실험/프로젝트는 현실적이지 않음

## 사전 준비

회의 시작 전 MUST 다섯 스킬 파일을 읽는다:
- `~/.claude/skills/goldie-pm/SKILL.md`
- `~/.claude/skills/goldie-po/SKILL.md`
- `~/.claude/skills/goldie-da/SKILL.md` ← **데이터 Single Source of Truth**
- `~/.claude/skills/goldie-devlead/SKILL.md`
- `~/.claude/skills/goldie-bible/SKILL.md` ← **도메인 지식 필수 참조** (전체를 읽되, 안건 관련 섹션을 회의에서 인용한다)

### 데이터 최신성 확인
- DA 스킬의 "최종 갱신일"을 확인한다
- 갱신일이 2주 이상 지났으면 → "⚠️ 데이터 갱신 필요" 경고를 회의 서두에 표시

### VOC 입력
- 회의 시 "이번 스프린트 상위 3개 CS 유형"을 PO가 입력한다
- CS 데이터가 없으면 → "VOC 미확인" 명시 후 진행

## 회의 절차

### 라운드 1 — 입장 표명

각 역할이 자기 스킬의 출력 형식대로 독립적으로 의견을 낸다.
다른 역할의 의견을 참고하지 않는다.

```
🟦 PM 의견
(goldie-pm 출력 형식)

🟩 PO 의견
(goldie-po 출력 형식)

🟧 DA 의견
(goldie-da 출력 형식)

🟥 Dev Lead 의견
(goldie-devlead 출력 형식)
```

### 라운드 2 — 반박

각 역할이 다른 두 역할에게 근거 기반으로 반박 또는 질문한다.

```
🟦 PM → 다른 역할: [반박/질문]
🟩 PO → 다른 역할: [반박/질문]
🟧 DA → 다른 역할: [반박/질문]
🟥 Dev Lead → 다른 역할: [반박/질문]
```

각 역할은 가장 이견이 큰 1~2개 역할에 집중 반박한다.

### 라운드 3 — 수정 입장

반박을 반영해 각 역할이 입장을 수정하거나 유지한다.

```
🟦 PM 수정 입장: [유지/수정 + 이유]
🟩 PO 수정 입장: [유지/수정 + 이유]
🟧 DA 수정 입장: [유지/수정 + 이유]
🟥 Dev Lead 수정 입장: [유지/수정 + 이유]
```

### 최종 결론

```
📋 최종 결론
━━━━━━━━━━━━━━━━━━━━━━━
합의 사항: [동의하는 부분]
미합의 사항: [의견 갈리는 부분 + 각 입장]
결정: [최종 추천 행동]
근거: [왜 이 결정인지]
반증 시나리오: [이 결정이 틀릴 경우 어떤 신호로 알 수 있는가]
실행 계획: [누가 / 며칠 / 어떻게]
다음 단계: [구체적 액션 아이템]
━━━━━━━━━━━━━━━━━━━━━━━
```

## 규칙

- 네 역할 모두 찬성 → MUST 놓친 관점 없는지 재확인
- DA가 "데이터 부족" → 소규모에서 가능한 검증 방법 먼저 설계
- Dev Lead가 "L/XL" 판정 → 스코프 축소 논의 MUST 선행
- 결론 불가 → "보류 + 필요 검증 사항"도 유효한 결론
- **실행 계획 없는 결론 금지** — 반드시 담당자와 일정 명시
- **Dev Lead가 불가 판정한 스코프를 그대로 결론 내는 것 금지**

### 의견 충돌 해소 원칙

역할 간 의견이 충돌할 때, 다음 우선순위로 판단한다:
1. **데이터 근거가 있는 쪽** 우선 (추측 < 수치)
2. 동등하면 **리스크가 낮은 쪽** 우선
3. 그래도 동등하면 **실행 속도가 빠른 쪽** 우선

### PO 긴급 안건

- 리텐션 핵심 지표(DAU, 재방문율)가 전주 대비 20% 이상 하락 시, PO가 긴급 안건을 상정할 수 있다
- 긴급 안건은 일반 우선순위와 별개로 즉시 논의한다
- 단, 기존 규칙(Dev Lead L/XL 판정, 실행 계획 필수 등)은 긴급 안건에도 동일하게 적용된다

## 검증 체크리스트

회의 완료 후 MUST 확인:
- [ ] 3라운드(입장→반박→수정) 모두 진행했는가?
- [ ] 각 역할이 자기 스킬의 출력 형식을 따랐는가?
- [ ] 반박이 감정이 아닌 근거 기반인가?
- [ ] 최종 결론에 실행 계획(누가/며칠)이 포함되었는가?
- [ ] 최종 결론에 반증 시나리오가 포함되었는가?
- [ ] 7명 팀으로 2주 내 실행 가능한 결론인가?
- [ ] 검증 방법이 소규모 모수에 맞는가?
- [ ] Bible(도메인 지식)을 참조했는가?
- [ ] 디자인 영향이 있으면 디자인 게이트(아래)를 통과했는가?
- [ ] 회귀/품질 리스크가 있으면 QA 게이트(아래)를 통과했는가?

하나라도 빠지면 → 해당 라운드 보완 후 재출력

## 게이트 추가 — Design / QA (2026-05-11 도입)

> gstack의 "CEO→Eng→Designer→QA 순차 게이트" 컨셉 차용.
> 4역할 토론(PM/PO/DA/DevLead) **이후**, 디자인·품질 영향이 있는 안건은 아래 2개 게이트를 *결론 확정 전*에 통과시킨다.
> 안건에 해당 영향이 없으면 게이트 생략 가능 (활성/생략 사유를 결론에 명시).

### 🎨 Design Gate

**활성 조건**: 화면·콘텐츠·DS·톤·시각 자산이 결과물에 포함되는 안건.

**검토 항목** (`~/.claude/skills/goldie-design-system/references/tone-matrix.md` 기준):
- 톤 매트릭스 lookup 결과가 명시되었는가? (위클리 하락 → 다크 외 케이스는 다크 커버 금지)
- DS 토큰 사용 100%? (atomic 색 직접 사용·#000·회색 bg 무 위반)
- 폰트·간격(4배수)·radius·shadow 표준 준수?
- 의도 1줄 선언이 있는가? ("왜 이 톤·이 컴포넌트인가")
- 접근성 (대비 4.5:1 이상, tabular-nums 등) 체크?

**산출물**:
```
🎨 Design Gate: PASS / FAIL
- 톤 결정: {유형} / Cover {nodeId} / Card {hex} / Accent {hex}
- 의도: "{1줄}"
- DS 위반 0건 / N건 ({목록})
- 다음 단계: {조채연·이예림 컨펌 요청 or 즉시 진행}
```

### 🛡️ QA Gate

**활성 조건**: 프로덕션 영향·데이터 무결성·결제·정책·외부 노출 변경이 포함되는 안건.

**검토 항목** (부대표 QA 정의서 + `goldie-qa-master` 에이전트 산출물 참조):
- 회귀 시나리오: 이 변경으로 깨질 수 있는 기존 기능 N개 + 검증 방법
- 데이터/시간 무결성: `goldie-time-keeper` 선행 통과 여부 (시간 작업 시 필수)
- 롤백 가능성: 실패 시 되돌릴 방법 + 소요 시간
- 모니터링 지표: 배포 후 24h/7d 관찰 대상 KPI
- 데드맨 스위치: 자동 정지 조건 (지표 N% 하락 등)

**산출물**:
```
🛡️ QA Gate: PASS / FAIL / CONDITIONAL
- 회귀 리스크: {N개 영역 + 검증 방법}
- 롤백: {가능/시간 N분}
- 모니터링: {지표 + 임계치}
- 다음 단계: {goldie-dev-qa watch 등록 or 안비손 부대표 컨펌}
```

### 게이트 운영 룰

1. 게이트는 4역할 라운드 3 *이후* 진행. 반박/수정에는 게이트 결과가 영향 미치지 않음 (별도 슬롯).
2. Design Gate FAIL → 결론 보류 + DS·톤 정정 후 재게이트
3. QA Gate FAIL → 결론 보류 + 회귀 시나리오 보강 또는 스코프 축소
4. 게이트 생략 시 결론에 *생략 사유* 명시 ("디자인 변경 없음" / "프로덕션 영향 없음")
5. 게이트 통과 후 결론에 *통과 결과*를 결론 표 마지막 줄로 첨부
